# RAG Implementation — Agentic Cloud Assistant

## Overview

Retrieval-Augmented Generation (RAG) was added to the Agentic Cloud Assistant
to address a specific limitation: before RAG, the LLM could answer questions
only from its general training memory and from live AWS scan data injected at
query time, but it had no access to structured security best-practice documents,
CIS Benchmark controls, or Terraform remediation patterns. This meant its
security advice was generic and uncitable. The RAG layer solves this by
maintaining a persistent ChromaDB vector store pre-loaded with curated AWS
security documentation; at query time it retrieves the most semantically
relevant chunks, injects them as labelled context into the prompt before the
user's question, and the LLM answers from that grounded material — citing
specific CIS control IDs and exact configuration values rather than producing
plausible-sounding but unsupported recommendations.

## Architecture

### Direct query path (user-facing `/rag/query` endpoint)

```
User question
    │
    ▼
POST /rag/query  (main.py)
    │
    ▼
query_knowledge_base()  (rag_service.py)
    │  ├─ knowledge_base.search()  (knowledge_base.py)
    │       └─ SentenceTransformer encodes question → query vector
    │           └─ ChromaDB cosine similarity search
    │               └─ top-N chunks returned with relevance scores
    │  ├─ RELEVANCE_THRESHOLD filter  (score >= 0.3)
    │  └─ Optional resource_type filter on metadata
    │
    ▼
build_augmented_prompt()  (rag_service.py)
    │  Retrieved chunks prepended before the question as labelled sources
    │
    ▼
chat_with_groq() / chat_with_anthropic() / chat_with_ollama()  (llm_service.py)
    │  Augmented prompt sent as the `message` argument; scan_data={}
    │
    ▼
Grounded answer returned to caller
    {question, answer, sources, chunks_used, raw_chunks}
```

### Agentic loop path (autonomous agent in `agent_service.py`)

```
agent_service.run_security_agent()
    │
    ▼
client.messages.create(tools=TOOLS, ...)   ← TOOLS includes RAG_TOOL_DEFINITION
    │
    │  Model emits tool_use block:
    │  { name: "query_security_knowledge_base", input: {query, resource_type} }
    │
    ▼
dispatch_tool(tool_name, tool_input)  (terraform_service.py)
    │
    ▼  tool_name == "query_security_knowledge_base"
    │
handle_rag_tool_call(tool_input)  (rag_service.py)
    │
    ▼
query_knowledge_base()  →  knowledge_base.search()  →  ChromaDB
    │
    ▼
Result dict returned to agentic loop:
    {knowledge_retrieved, sources_consulted, chunks_retrieved, context}
    │
    ▼
Packaged as tool_result message → fed back to model in next turn
```

## File Structure

```
backend/rag/__init__.py
    Empty package marker — makes rag/ a Python package importable as
    `from rag.knowledge_base import ...`

backend/rag/knowledge_base.py
    Core storage and retrieval engine. Defines SecurityKnowledgeBase class
    with chunk_text(), add_document(), search(), delete_document(), and
    list_documents(). Holds the module-level singleton `knowledge_base`.
    Constants: CHUNK_SIZE=400, CHUNK_OVERLAP=50, EMBEDDING_MODEL_NAME,
    CHROMA_DB_PATH, COLLECTION_NAME.

backend/rag/seed_knowledge.py
    One-time setup script that ingests 5 curated AWS security documents into
    ChromaDB. Idempotent — uses upsert so it is safe to run multiple times.
    Run with: cd backend && python -m rag.seed_knowledge

backend/rag/rag_service.py
    Query-time logic layer. Defines query_knowledge_base(), which calls
    knowledge_base.search(), applies RELEVANCE_THRESHOLD filtering and
    optional resource_type filtering, then calls build_augmented_prompt().
    Also defines RAG_TOOL_DEFINITION (MCP tool schema) and
    handle_rag_tool_call() (the dispatch target).

frontend/src/components/KnowledgeBasePanel.jsx
    React panel with two tabs: Query (search the knowledge base and display
    a grounded LLM answer with source chips and collapsible chunk viewer)
    and Manage Documents (upload PDF/txt files, paste raw text, list all
    documents with chunk counts, delete documents). Mounted in Dashboard.jsx
    as a full-width panel. Reads API keys from useApiKeys() context.

chroma_db/
    Persistent on-disk ChromaDB store. Created automatically on first run
    of knowledge_base.py. Contains chroma.sqlite3 (collection metadata and
    document records) and a UUID-named directory (HNSW vector index files).
    Not committed to version control — regenerated by running seed_knowledge.py.
```

## How It Works — Indexing Phase

When a document is added (either via `seed_knowledge.py` or a user upload):

1. **Text arrives** at `knowledge_base.add_document(doc_id, text, metadata)`.

2. **`chunk_text(text)`** splits the text by whitespace into word tokens, then
   slides a window of `CHUNK_SIZE=400` words across the document with a step
   of `CHUNK_SIZE - CHUNK_OVERLAP = 350` words. This produces overlapping
   chunks so that no sentence is silently cut at a boundary.

3. **`self.encoder.encode(chunks)`** calls the lazily-loaded
   `SentenceTransformer("all-MiniLM-L6-v2")` model to convert each chunk
   string into a 384-float vector. The model is loaded on the first call and
   reused for all subsequent calls (singleton pattern on `self._encoder`).

4. **Chunk IDs** are constructed as `"{doc_id}__chunk_{i}"` — e.g.
   `"aws-s3-security__chunk_0"`. This naming convention allows
   `delete_document()` to find all chunks for a given document by prefix scan.

5. **Per-chunk metadata** is assembled: the caller-supplied fields (e.g.
   `source`, `resource_type`, `severity`) are merged with auto-generated
   fields `doc_id`, `chunk_index`, and `total_chunks`.

6. **`self.collection.upsert(ids, embeddings, documents, metadatas)`** writes
   all chunks to ChromaDB atomically. Upsert means existing chunks are
   overwritten rather than duplicated — safe to re-run.

7. ChromaDB persists the HNSW vector index to `chroma_db/` on disk. The data
   survives backend restarts without re-embedding.

## How It Works — Query Phase

When a question arrives at `POST /rag/query` or via the agentic tool call:

1. **`query_knowledge_base(query, n_results, resource_filter)`** in
   `rag_service.py` is the entry point.

2. **`knowledge_base.search(query, n_results=3)`** encodes the question text
   into a vector using the same `all-MiniLM-L6-v2` model, then calls
   `self.collection.query(query_embeddings=..., n_results=...,
   include=["documents", "metadatas", "distances"])`.

3. ChromaDB returns up to `n_results` chunks sorted by cosine distance.
   **`relevance_score` is computed as `1 - distance`** (ChromaDB returns
   cosine distances in [0, 2]; a distance of 0.3 becomes a relevance of 0.7).

4. **`RELEVANCE_THRESHOLD = 0.3` filter** discards any chunk whose
   `relevance_score < 0.3`. Chunks with very low scores add noise to the
   prompt rather than grounding it.

5. If **`resource_filter`** was provided (e.g. `"s3"`), chunks whose
   `metadata["resource_type"]` does not match are discarded, keeping the
   prompt focused on the relevant AWS service.

6. **`build_augmented_prompt(query, chunks)`** constructs the final LLM
   prompt. Each surviving chunk is formatted as:
   ```
   [Source N: {doc_id} | relevance: {score}]
   {chunk text}
   ```
   All chunks are joined with `---` separators, placed *before* the question,
   and wrapped in a system-role framing as an AWS security expert.

7. The augmented prompt is passed as the `message` argument to
   `chat_with_groq()`, `chat_with_anthropic()`, or `chat_with_ollama()` in
   `llm_service.py` with `scan_data={}` (no AWS scan needed for RAG queries).

8. The LLM reply, source doc IDs, chunk count, and raw chunks are returned to
   the caller.

## API Endpoints

| Method | Path                        | Purpose                                                                       |
|--------|-----------------------------|-------------------------------------------------------------------------------|
| POST   | `/rag/query`                | Retrieve relevant chunks and answer a question with an LLM grounded in retrieved context. Accepts `question`, `resource_type`, `n_results`, `groq_key`, `model_provider`. |
| POST   | `/rag/documents/upload`     | Upload a PDF or plain-text file as `multipart/form-data`. Extracts text via PyPDF2 for PDFs. Accepts `file`, `doc_id`, `resource_type`, `source` form fields. |
| POST   | `/rag/documents/text`       | Ingest a plain-text document from a JSON body. Accepts `doc_id`, `text`, `resource_type`, `source`, `title`. |
| GET    | `/rag/documents`            | List all documents in the knowledge base. Returns `documents` array (each with `doc_id`, `source`, `resource_type`, `chunk_count`), `total_documents`, and `total_chunks`. |
| DELETE | `/rag/documents/{doc_id}`   | Delete a document and all its chunks from ChromaDB. Returns `status`, `deleted_chunks`, `doc_id`. |

All five endpoints are defined in `backend/main.py`. All errors are returned
as structured JSON — none raise HTTP 500.

## MCP Tool Integration

**Tool name:** `query_security_knowledge_base`

**Definition location:** `RAG_TOOL_DEFINITION` dict in
`backend/rag/rag_service.py` (lines 167–204). The dict has the three required
MCP fields: `name`, `description`, and `input_schema`. The schema declares
three properties: `query` (required string), `resource_type` (optional enum),
and `n_results` (optional integer, default 3).

**Registration:** `backend/services/terraform_service.py` imports
`RAG_TOOL_DEFINITION` at line 9:
```python
from rag.rag_service import RAG_TOOL_DEFINITION, handle_rag_tool_call
```
It is appended to the `TOOLS` list at line 85:
```python
TOOLS = [TERRAFORM_TOOL, RUN_PLAN_TOOL, SUMMARISE_PLAN_TOOL, RAG_TOOL_DEFINITION]
```
This list is imported by `agent_service.py` (`from services.terraform_service
import TOOLS, dispatch_tool`) and passed to every `client.messages.create()`
call, making the tool visible to the Claude model during the agentic loop.

**Dispatch:** When the model returns a `tool_use` block with
`name == "query_security_knowledge_base"`, `agent_service.py` calls
`dispatch_tool(tool_name, tool_input)`. The `dispatch_tool` function in
`terraform_service.py` routes it via:
```python
elif tool_name == "query_security_knowledge_base":
    return handle_rag_tool_call(tool_input)
```
`handle_rag_tool_call()` in `rag_service.py` calls `query_knowledge_base()`,
then returns `{knowledge_retrieved, sources_consulted, chunks_retrieved,
context}`. This dict is JSON-serialised and fed back to the model as a
`tool_result` message in the next conversation turn.

## Configurable Constants

| Constant               | Location                | Current Value        | What it controls                                                                                   |
|------------------------|-------------------------|----------------------|----------------------------------------------------------------------------------------------------|
| `CHUNK_SIZE`           | `knowledge_base.py:48`  | `400`                | Maximum number of words per chunk. Larger values capture more context per chunk but may exceed the embedding model's token limit and reduce retrieval precision. |
| `CHUNK_OVERLAP`        | `knowledge_base.py:49`  | `50`                 | Number of words shared between consecutive chunks. Prevents ideas from being silently split at a boundary. Step size = CHUNK_SIZE − CHUNK_OVERLAP = 350 words. |
| `RELEVANCE_THRESHOLD`  | `rag_service.py:36`     | `0.3`                | Minimum cosine similarity score (computed as `1 − distance`) required for a chunk to be included in the augmented prompt. Chunks below this value are discarded before the LLM sees them. |
| `EMBEDDING_MODEL_NAME` | `knowledge_base.py:47`  | `"all-MiniLM-L6-v2"` | HuggingFace sentence-transformers model used for both indexing and query encoding. Produces 384-dimensional vectors. Must be the same model at index time and query time. |

## Known Limitations

**1. Chunking is word-count-based, not token-aware.**
`chunk_text()` splits on whitespace and counts words, but `all-MiniLM-L6-v2`
has a maximum input of 256 word-piece tokens. A 400-word chunk that contains
long compound words or code identifiers may exceed 256 tokens and be silently
truncated by the tokeniser before encoding, producing a vector that represents
only the first part of the chunk. The fix would be to use the model's own
tokeniser to measure chunk size.

**2. All 5 seed documents each produce exactly 1 chunk.**
The seed texts were written to be concise summaries and each comes in under
400 words, so no splitting occurs. This means a query for a specific CIS
control number (e.g. "CIS 2.1.1") retrieves the entire S3 document rather
than a targeted paragraph. As the knowledge base grows with longer uploaded
documents, retrieval precision will improve.

**3. The document list is reconstructed by scanning all chunk metadata on every
`GET /rag/documents` call.**
`list_documents()` calls `self.collection.get(include=["metadatas"])`, which
fetches the metadata of every chunk in the collection and groups them by
`doc_id` in Python. This is acceptable at small scale but will become slow as
the collection grows to thousands of chunks — a separate documents table or
cached manifest would be needed for production use.

**4. Deletion by ID prefix scan is O(n) in the number of total chunks.**
`delete_document()` fetches all IDs in the collection with
`self.collection.get()` and filters in Python for those starting with
`{doc_id}__chunk_`. ChromaDB does not expose a native prefix-delete or
secondary-index query, so this scales linearly with collection size.

**5. `relevance_score = 1 - distance` is an approximation, not a true
probability.**
ChromaDB returns cosine distances in the range [0, 2]. The transformation
`1 - distance` maps that to [-1, 1], but for normalised sentence-transformer
embeddings the distances in practice fall in [0, 1], making the scores
interpretable as [0, 1]. However, this is a convention, not a calibrated
confidence value — a score of 0.7 means "more similar than 0.3" but does not
mean "70% probability of being the correct answer".
